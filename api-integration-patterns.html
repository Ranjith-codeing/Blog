API Integration Patterns in Business Central

A practical guide to building reliable integrations using Business Central's standard and custom APIs — covering OAuth, custom endpoints, webhooks, idempotency, and resilient error handling.

Ranjith · March 2026

Business Central exposes a rich REST API surface — standard OData endpoints for most entities, custom API pages for bespoke needs, and webhook subscriptions for event-driven patterns. Knowing which to use, and how to secure them properly, separates brittle integrations from ones that run for years without intervention.

Types of BC APIs

Business Central offers three categories of API surface, each suited to different scenarios:

TypeWhen to UseBase URL Pattern
Standard API v2.0 Reading/writing standard entities (customers, vendors, items, invoices) /api/v2.0/companies({id})/customers
Custom API Pages Business-specific entities not in v2.0, or combined views /api/ranjith/bc/v1.0/companies({id})/customEntity
OData v4 Web Services Legacy integrations, Power BI, Excel connections /ODataV4/Company('{name}')/PageName
💡
Recommendation: Prefer the v2.0 API for standard entities. Use custom API pages only for entities genuinely not covered. Avoid new OData Web Service integrations — they lack versioning and are harder to deprecate.

Authentication & OAuth 2.0

Business Central's API uses OAuth 2.0 exclusively for cloud environments. The two flows you'll encounter most are Client Credentials (service-to-service, no user) and Authorization Code (user-delegated access). For backend integrations, always use Client Credentials.

Acquiring a Token (Client Credentials Flow)

POST https://login.microsoftonline.com/{tenantId}/oauth2/v2.0/token

Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id={appClientId}
&client_secret={appClientSecret}
&scope=https://api.businesscentral.dynamics.com/.default

The response contains an access_token valid for 3600 seconds. Pass it as a Bearer token on every API request:

GET https://api.businesscentral.dynamics.com/v2.0/{tenantId}/{environment}/api/v2.0/companies
Authorization: Bearer {access_token}
Accept: application/json

Required Azure AD App Permissions

PermissionTypePurpose
Financials.ReadWrite.AllApplicationFull API read/write access
app_accessApplicationRequired for client credentials flow in BC
⚠️
After adding permissions in Azure AD, you must also add the app registration as an Allowed OAuth 2.0 App inside Business Central (under Azure Active Directory Applications) and assign it the correct BC permission sets.

Building a Custom API Page

When the standard v2.0 endpoints don't expose what you need, build a custom API page in AL. The page type is API, and BC handles the REST routing automatically — you define the entity, fields, and sub-pages. 

page 50300 "Sales Order Upload API"
{
    PageType    = API;
    APIPublisher = 'ranjith';
    APIGroup    = 'bc';
    APIVersion  = 'v1.0';
    EntityName  = 'salesOrderUpload';
    EntitySetName = 'salesOrderUploads';
    SourceTable = "Sales Order Upload Buffer";
    DelayedInsert = true;
    ODataKeyFields = SystemId;

    layout
    {
        area(Content)
        {
            repeater(Group)
            {
                field(id;              Rec.SystemId)            { }
                field(batchNo;         Rec."Batch No.")          { }
                field(customerNo;      Rec."Customer No.")       { }
                field(externalDocNo;   Rec."External Doc. No.")  { }
                field(postingDate;     Rec."Posting Date")       { }
                field(status;          Rec.Status)               { }
                field(errorMessage;    Rec."Error Message")      { }
                field(postedDocNo;     Rec."Posted Doc. No.")    { }
            }
        }
    }

    trigger OnInsertRecord(BelowxRec: Boolean): Boolean
    var
        Validator: Codeunit "Sales Order Upload Validator";
    begin
        // Validate immediately on insert
        Validator.ValidateRecord(Rec);
        exit(true);
    end;
}

Once published, the endpoint becomes:

GET  /api/ranjith/bc/v1.0/companies({companyId})/salesOrderUploads
POST /api/ranjith/bc/v1.0/companies({companyId})/salesOrderUploads
GET  /api/ranjith/bc/v1.0/companies({companyId})/salesOrderUploads({id})

Calling External APIs from AL

AL's HttpClient lets you call any external REST API. The pattern is consistent: build the request, send it, check the status code, parse the response. Always wrap in a [TryFunction] for graceful failure. 

codeunit 50301 "External API Client"
{
    procedure GetExchangeRate(CurrencyCode: Code[10]): Decimal
    var
        Client:   HttpClient;
        Request:  HttpRequestMessage;
        Response: HttpResponseMessage;
        Headers:  HttpHeaders;
        JsonObj:  JsonObject;
        RateToken: JsonToken;
        ResponseText: Text;
        Rate: Decimal;
    begin
        Request.Method := 'GET';
        Request.SetRequestUri(
            'https://api.exchangeratesapi.io/v1/latest?base=EUR&symbols=' + CurrencyCode);

        Request.GetHeaders(Headers);
        Headers.Add('Accept', 'application/json');
        Headers.Add('Authorization', 'Bearer ' + GetApiKey());

        if not Client.Send(Request, Response) then
            Error('Exchange rate API call failed — network error.');

        if not Response.IsSuccessStatusCode() then
            Error('Exchange rate API returned status %1.', Response.HttpStatusCode());

        Response.Content.ReadAs(ResponseText);
        JsonObj.ReadFrom(ResponseText);

        if JsonObj.SelectToken('rates.' + CurrencyCode, RateToken) then
            Rate := RateToken.AsValue().AsDecimal()
        else
            Error('Currency code %1 not found in response.', CurrencyCode);

        exit(Rate);
    end;

    local procedure GetApiKey(): Text
    var
        Key: Text;
    begin
        if not IsolatedStorage.Get('ExchangeRateApiKey', DataScope::Company, Key) then
            Error('Exchange rate API key is not configured.');
        exit(Key);
    end;
}

Webhooks & Event-Driven Patterns

Instead of polling BC every few minutes, webhook subscriptions push notifications to your external system the moment something changes. BC supports webhooks on any API entity via the subscriptions endpoint.

Creating a Webhook Subscription

POST /api/v2.0/subscriptions

{
  "notificationUrl": "https://your-service.com/bc-webhook",
  "resource":        "/api/v2.0/companies({id})/salesInvoices",
  "clientState":     "your-secret-validation-token"
}

BC will immediately send a validation request to your notificationUrl. Your endpoint must echo back the validationToken query parameter as plain text with a 200 OK response.

Notification Payload

{
  "value": [
    {
      "subscriptionId": "abc123",
      "clientState":    "your-secret-validation-token",
      "changeType":     "updated",
      "resource":       "/api/v2.0/companies({id})/salesInvoices({invoiceId})",
      "resourceData": {
        "id": "{invoiceId}"
      }
    }
  ]
}
💡
Always verify clientState on every incoming notification before processing. This prevents replay attacks from unauthenticated callers. Treat any notification with a mismatched clientState as invalid and return 401.

Renewing Subscriptions

Webhook subscriptions expire after 3 days by default. Build a scheduled renewal — a Job Queue Entry that runs daily and PATCHes each subscription's expiry date forward:

PATCH /api/v2.0/subscriptions({subscriptionId})

{
  "notificationUrl": "https://your-service.com/bc-webhook"
}

Idempotency & Safe Retries

Network failures happen. Your integration must handle the same event being delivered more than once without creating duplicate records. Two patterns work well:

Pattern 1 — Idempotency Key

Pass a unique key with every write operation. If you've already processed that key, return the cached result instead of re-executing:

// Store processed keys with their result
table 50302 "Idempotency Log"
{
    fields
    {
        field(1; "Idempotency Key"; Text[100]) { }
        field(2; "Result";          Text[250]) { }
        field(3; "Created At";      DateTime)  { }
    }
    keys { key(PK; "Idempotency Key") { Clustered = true; } }
}

// Before processing any inbound request:
procedure IsAlreadyProcessed(Key: Text; var Result: Text): Boolean
var
    Log: Record "Idempotency Log";
begin
    if Log.Get(Key) then begin
        Result := Log.Result;
        exit(true);
    end;
    exit(false);
end;

Pattern 2 — External Doc. No. Uniqueness Check

For invoice creation, check External Document No. uniqueness per customer before inserting. If it already exists, return the existing BC document number instead of failing or creating a duplicate.

Error Handling Patterns

01

Structured Error Responses

Custom API pages should return JSON error bodies, not plain text. Use Error() with meaningful messages — BC wraps them in the standard OData error envelope automatically.

02

Exponential Back-off

On 429 Too Many Requests or 503, wait and retry. Start at 500ms, double each attempt. Cap at 3 retries before alerting operations.

03

Dead Letter Queue

Any webhook notification that fails processing after 3 retries should land in a dead-letter table. Operations can inspect and manually re-trigger without losing the event.

04

Correlation IDs

Log the x-ms-correlation-request-id header from every BC API response. When raising a Microsoft support ticket, this header is essential for tracing the server-side log.

Production Readiness Checklist

OAuth token cached and refreshed before expiry — no per-request token calls
Client Secret stored in Isolated Storage, not in setup table fields
Webhook clientState validated on every inbound notification
Webhook subscriptions renewed daily via Job Queue
Idempotency keys or External Doc. No. uniqueness enforced
Exponential back-off implemented for 429 / 503 responses
Dead-letter table for failed webhook events
Correlation IDs logged for every API response
Custom API page versioned (v1.0) — never modify a live API's fields
