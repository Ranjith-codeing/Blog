Business Central Telemetry with Application Insights

When a customer says "it was slow yesterday afternoon," guessing is not a strategy. Telemetry turns that into a query. Here's how to instrument an extension and read the signals.

R
Ranjith · April 2026

Business Central emits a rich stream of telemetry — page loads, report runs, web service calls, lock timeouts, even long-running SQL — to Azure Application Insights. On top of that, your own extension can emit custom signals from AL with a single call. Once it's flowing, "the integration broke last night" becomes a KQL query with a timestamp, an error message, and a stack of custom dimensions. This is how you stop debugging by anecdote.

Why telemetry, not logging

A custom "log table" in BC has three problems: it bloats the database you're trying to keep fast, it's invisible to anyone without BC access, and it disappears on a sandbox refresh. Application Insights solves all three — the data lives in Azure, it's queryable with KQL, it survives environment resets, and it costs you nothing in BC table size. It's the production complement to the performance work you do at design time.

Connecting Application Insights

You wire telemetry at one (or both) of two scopes. Modern BC uses the Application Insights connection string, not the legacy instrumentation key alone.

  1. Create an Application Insights resource in Azure and copy its Connection String.
  2. Environment scope — in the Business Central admin center, set the connection string on the environment. This captures platform + all extensions' telemetry.
  3. Extension scope — set it in app.json so your app's signals flow to your resource even on a customer tenant you don't administer.
{
  "id": "a1b2c3d4-....",
  "name": "CIS Connector",
  "publisher": "Ranjith",
  "version": "1.0.0.0",
  "application": "24.0.0.0",
  "applicationInsightsConnectionString": "InstrumentationKey=xxxx;IngestionEndpoint=https://...applicationinsights.azure.com/"
}
⚠️
Don't commit a real connection string to source control. Inject it at build time from a pipeline secret (see the DevOps notes in API Patterns), or set it per-environment in the admin center instead of app.json.

Emitting Custom Signals

The whole API is one method: Session.LogMessage. You give it a stable event ID, a message, a verbosity, a data classification, a scope, and a dictionary of custom dimensions. Those dimensions are the gold — they're what you filter and group on later. 

codeunit 50800 "CIS Telemetry"
{
    procedure LogVerification(VendorNo: Code[20]; Treatment: Text;
                              DurationMs: Integer; Success: Boolean)
    var
        Dimensions: Dictionary of [Text, Text];
        Verbosity: Verbosity;
    begin
        Dimensions.Add('vendorNo', VendorNo);
        Dimensions.Add('taxTreatment', Treatment);
        Dimensions.Add('durationMs', Format(DurationMs));
        Dimensions.Add('outcome', Format(Success));

        if Success then
            Verbosity := Verbosity::Normal
        else
            Verbosity := Verbosity::Error;

        // EventId: a stable, greppable code you own — never reuse it for a different signal
        Session.LogMessage(
            'CIS0001',
            'HMRC subcontractor verification completed',
            Verbosity,
            DataClassification::SystemMetadata,
            TelemetryScope::ExtensionPublisher,
            Dimensions);
    end;
}
ArgumentWhat it controls
eventIdYour stable identifier — the primary thing you query on. Prefix it (CIS0001) so it's unmistakably yours.
verbosityCritical · Error · Warning · Normal · Verbose — lets you filter noise from incidents.
dataClassificationTags the message content; keep it SystemMetadata and put no PII in the message.
telemetryScopeExtensionPublisher (your resource only) or All (also the environment resource).

Designing Good Dimensions

The message string is for humans; the custom dimensions are for queries. A few rules that pay off later:

Put identifiers and measures in dimensions (vendorNo, durationMs), not baked into the message text
Keep dimension names stable across versions — renaming one breaks every saved query and alert
Use a consistent eventId scheme so a glob like CIS* finds all your signals
Never log UTRs, tokens, emails, or other PII — telemetry is not access-controlled like BC data

Reading Signals with KQL

All of it lands in the traces table. BC's own signals and yours sit side by side, distinguished by the eventId custom dimension. Start by finding your events: 

traces
| where timestamp > ago(24h)
| where customDimensions.eventId == "CIS0001"
| project timestamp,
          vendorNo   = tostring(customDimensions.vendorNo),
          treatment  = tostring(customDimensions.taxTreatment),
          ms         = toint(customDimensions.durationMs),
          outcome    = tostring(customDimensions.outcome)
| order by timestamp desc

Now turn raw rows into an answer — the 95th-percentile verification latency per day, and the failure rate:

traces
| where customDimensions.eventId == "CIS0001"
| extend ms = toint(customDimensions.durationMs),
         ok = tostring(customDimensions.outcome) == "Yes"
| summarize
      calls        = count(),
      failures     = countif(not ok),
      p95_latency  = percentile(ms, 95)
  by bin(timestamp, 1d)
| extend failure_rate = round(100.0 * failures / calls, 1)
| order by timestamp desc

And to catch the platform telling you about slow SQL behind your own feature, lean on BC's built-in signals — for example long-running operations:

traces
| where timestamp > ago(7d)
| where customDimensions.eventId == "RT0018"   // long running AL method (platform signal)
| extend objectName = tostring(customDimensions.alObjectName),
         ms         = toint(customDimensions.executionTimeInMs)
| where objectName startswith "CIS"
| summarize avg(ms), max(ms), count() by objectName
| order by max_ms desc
💡
You don't have to invent every query. Microsoft's open-source BCTech repository ships a large library of ready-made KQL for BC telemetry — page performance, web service latency, errors, company-by-company breakdowns. Start there and adapt.

Dashboards & Alerts

01

Azure Workbooks

Pin your KQL into an Azure Workbook for a living dashboard — latency trends, failure rates, error counts per company. Share it with the customer's IT team read-only.

02

Alert Rules

Create a log-based alert on a query like "more than 5 CIS0001 failures in 15 minutes" and route it to Teams or email. You hear about breakage before the customer calls.

03

Release Health

Group any signal by customDimensions.appVersion to compare error rates before and after a deployment — your fastest regression detector.

04

Multi-Tenant View

With publisher-scope telemetry across customers, group by aadTenantId to see which customer is hitting an issue — without logging into any of their environments.

Best Practices

Instrument the seams — integration calls, posting, long loops, anything that talks to the outside world
Emit a duration on every external call so latency is queryable from day one
Use Verbosity::Verbose for chatty signals so you can filter them out cheaply
Document your event IDs in the repo — a small table of code → meaning
Don't emit telemetry inside tight per-record loops — sample or aggregate first

Conclusion

Telemetry is the cheapest insurance an extension can carry. One Session.LogMessage per meaningful event, a handful of well-named dimensions, and a few saved KQL queries give you production visibility that a log table never will. Instrument the seams, never log PII, and let the data answer the questions you'd otherwise be guessing at.

Key takeaways
  • Set the Application Insights connection string at environment and/or extension scope.
  • Emit custom signals with one call: Session.LogMessage with a stable eventId.
  • Put identifiers and measures in custom dimensions — that's what you query.
  • Everything lands in traces; read it with KQL, reuse the BCTech library.
  • Never log PII. Telemetry isn't access-controlled like BC data.
