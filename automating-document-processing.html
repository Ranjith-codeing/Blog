Automating Document Processing with Business Central

Using AL event subscribers, Job Queue scheduling, approval workflows, and Power Automate triggers to eliminate manual touchpoints across purchase, sales, and finance document flows.

Ranjith · March 2026

Every time a user manually opens a document, checks a field, and clicks Post — that's a process waiting to be automated. Business Central provides a rich event system, a built-in scheduler, and deep Power Automate integration. This guide connects those pieces into working automation patterns.

The Event-Driven Model

Business Central's event system is the backbone of all automation. Rather than modifying base objects directly (which breaks upgrades), you subscribe to published events on standard codeunits and tables. When BC triggers those events, your subscriber code runs automatically.

📄
Document Action
User posts, releases, or creates a document
→
📡
BC Publishes Event
OnAfterPost, OnAfterRelease, OnAfterInsert…
→
⚙️
Your Subscriber Runs
Notification, linking, validation, or queuing

The three event categories you'll use most:

CategoryExample EventsTypical Use
Business EventsOnAfterPostSalesInvoice, OnAfterReleasePurchaseDocTriggering downstream actions after document state changes
Integration EventsOnBeforeInsertPostedPurchInvCopying custom fields to posted tables, enriching data
Table TriggersOnAfterInsertEvent, OnAfterModifyEventReacting to any record change without modifying the table

Event Subscribers in AL

An event subscriber is a regular AL procedure decorated with the [EventSubscriber] attribute. It lives in any codeunit with EventSubscriberInstance = StaticAutomatic — BC instantiates and calls it automatically. 

codeunit 50500 "Document Automation Subscribers"
{
    // StaticAutomatic = BC creates an instance automatically when the event fires
    SingleInstance = false;

    // Fires after a Sales Invoice is posted
    [EventSubscriber(ObjectType::Codeunit, Codeunit::"Sales-Post",
                     'OnAfterFinalizePosting', '', false, false)]
    local procedure OnAfterSalesInvoicePosted(
        var SalesHeader: Record "Sales Header";
        var SalesShipmentHeader: Record "Sales Shipment Header";
        var SalesInvoiceHeader: Record "Sales Invoice Header";
        var SalesCrMemoHeader: Record "Sales Cr.Memo Header";
        var ReturnReceiptHeader: Record "Return Receipt Header";
        var GenJnlPostLine: Codeunit "Gen. Jnl.-Post Line";
        CommitIsSuppressed: Boolean)
    var
        Notifier: Codeunit "Document Notification Manager";
        Archiver: Codeunit "Document Archive Manager";
    begin
        if SalesInvoiceHeader."No." = '' then exit;

        // Queue email notification (never block posting with synchronous sends)
        Notifier.QueueInvoiceNotification(SalesInvoiceHeader."No.");

        // Archive to SharePoint
        Archiver.ScheduleArchive('SalesInvoice', SalesInvoiceHeader."No.");
    end;

    // Fires after a Purchase Order is released
    [EventSubscriber(ObjectType::Codeunit, Codeunit::"Release Purchase Document",
                     'OnAfterReleasePurchaseDoc', '', false, false)]
    local procedure OnAfterPurchaseOrderReleased(var PurchaseHeader: Record "Purchase Header")
    var
        ApprovalMgr: Codeunit "Document Approval Manager";
    begin
        if PurchaseHeader."Document Type" <> PurchaseHeader."Document Type"::Order then
            exit;

        // Trigger approval flow for orders above threshold
        if PurchaseHeader.Amount > GetApprovalThreshold() then
            ApprovalMgr.RequestApproval(PurchaseHeader);
    end;

    local procedure GetApprovalThreshold(): Decimal
    var
        Setup: Record "Automation Setup";
    begin
        if Setup.Get() then
            exit(Setup."PO Approval Threshold");
        exit(5000); // default
    end;
}
⚠️
Never call slow operations directly in an event subscriber. Sending emails, calling APIs, or running heavy queries inside a subscriber adds to the user's posting wait time. Always enqueue work via Job Queue instead.

Automatic Document Posting

Certain document types — recurring journals, auto-invoices from confirmed orders, or scheduled standing charges — should post on a schedule without human intervention. The Job Queue makes this reliable and auditable. 

codeunit 50501 "Auto Post Recurring Invoices"
{
    TableNo = "Job Queue Entry";

    trigger OnRun()
    var
        SalesHeader: Record "Sales Header";
        PostingMgr:  Codeunit "Sales-Post";
        LogEntry:    Record "Auto Post Log";
        ErrorText:   Text;
    begin
        // Find all invoices flagged for automatic posting today
        SalesHeader.SetRange("Document Type", SalesHeader."Document Type"::Invoice);
        SalesHeader.SetRange("Auto Post", true);
        SalesHeader.SetRange("Posting Date", Today());
        SalesHeader.SetRange(Status, SalesHeader.Status::Released);

        if not SalesHeader.FindSet() then exit;

        repeat
            Commit(); // isolate each invoice

            Clear(ErrorText);
            if not TryPostSingle(SalesHeader, PostingMgr) then
                ErrorText := GetLastErrorText();

            // Log every attempt
            LogEntry.Init();
            LogEntry."Document No."   := SalesHeader."No.";
            LogEntry."Customer No."   := SalesHeader."Sell-to Customer No.";
            LogEntry."Attempted At"   := CurrentDateTime();
            LogEntry.Success          := ErrorText = '';
            LogEntry."Error Message"  := ErrorText;
            LogEntry.Insert();

        until SalesHeader.Next() = 0;
    end;

    [TryFunction]
    local procedure TryPostSingle(var SalesHeader: Record "Sales Header";
                                  var PostingMgr: Codeunit "Sales-Post")
    begin
        PostingMgr.Run(SalesHeader);
    end;
}

Approval Workflow Integration

Business Central's built-in Approval Workflow engine handles multi-level approvals with delegations, substitutions, and due-date escalations. You can trigger it programmatically from AL or configure it through the standard Workflow setup UI.

Triggering an Approval Request from AL

codeunit 50502 "Document Approval Manager"
{
    procedure RequestApproval(var PurchaseHeader: Record "Purchase Header")
    var
        ApprovalsMgmt: Codeunit "Approvals Mgmt.";
    begin
        // Validate that a workflow is configured for this document type
        if not ApprovalsMgmt.CheckPurchaseApprovalPossible(PurchaseHeader) then
            Error('No approval workflow is configured for Purchase Orders. Contact your administrator.');

        // Send the approval request — BC handles routing to the correct approver
        ApprovalsMgmt.OnSendPurchaseDocForApproval(PurchaseHeader);

        // Update document status to Pending Approval
        PurchaseHeader.Status := PurchaseHeader.Status::"Pending Approval";
        PurchaseHeader.Modify(true);
    end;

    procedure CancelApproval(var PurchaseHeader: Record "Purchase Header")
    var
        ApprovalsMgmt: Codeunit "Approvals Mgmt.";
    begin
        ApprovalsMgmt.OnCancelPurchaseApprovalRequest(PurchaseHeader);
    end;
}

Custom Approval Conditions

// Subscribe to the approval check event to inject custom routing logic
[EventSubscriber(ObjectType::Codeunit, Codeunit::"Approvals Mgmt.",
                 'OnPopulateApprovalEntryArgument', '', false, false)]
local procedure SetCustomApprover(
    var RecRef: RecordRef;
    var ApprovalEntryArgument: Record "Approval Entry")
var
    PurchHeader: Record "Purchase Header";
    VendorCategory: Code[20];
begin
    if RecRef.Number <> Database::"Purchase Header" then exit;

    RecRef.SetTable(PurchHeader);
    VendorCategory := GetVendorCategory(PurchHeader."Buy-from Vendor No.");

    // Route to Finance Director for strategic vendors
    if VendorCategory = 'STRATEGIC' then
        ApprovalEntryArgument."Approver ID" := GetFinanceDirectorUserId();
end;

Email & Teams Notifications

Business Central's Email module (introduced in 2021 wave 1) replaces the old SMTP setup with a connector-based model. Sending a notification from AL is now straightforward and testable. 

codeunit 50503 "Document Notification Manager"
{
    procedure QueueInvoiceNotification(InvoiceNo: Code[20])
    var
        JobQueueEntry: Record "Job Queue Entry";
    begin
        // Queue the send — don't block the caller's session
        JobQueueEntry.Init();
        JobQueueEntry."Object Type to Run"   := JobQueueEntry."Object Type to Run"::Codeunit;
        JobQueueEntry."Object ID to Run"     := Codeunit::"Invoice Email Sender";
        JobQueueEntry."Parameter String"     := InvoiceNo;
        JobQueueEntry."Earliest Start Date/Time" := CurrentDateTime() + 30000; // 30 sec delay
        JobQueueEntry.Insert(true);
        CODEUNIT.Run(CODEUNIT::"Job Queue - Enqueue", JobQueueEntry);
    end;
}

codeunit 50504 "Invoice Email Sender"
{
    TableNo = "Job Queue Entry";

    trigger OnRun()
    var
        InvoiceNo:   Code[20];
        Invoice:     Record "Sales Invoice Header";
        Email:       Codeunit Email;
        Message:     Codeunit "Email Message";
        Customer:    Record Customer;
        Body:        Text;
    begin
        InvoiceNo := CopyStr(Rec."Parameter String", 1, 20);

        if not Invoice.Get(InvoiceNo) then exit;
        if not Customer.Get(Invoice."Sell-to Customer No.") then exit;
        if Customer."E-Mail" = '' then exit;

        Body := BuildEmailBody(Invoice);

        Message.Create(Customer."E-Mail",
                        'Invoice ' + InvoiceNo + ' from ' + CompanyName(),
                        Body,
                        true); // isHtml

        Email.Send(Message, Enum::"Email Scenario"::Default);
    end;

    local procedure BuildEmailBody(Invoice: Record "Sales Invoice Header"): Text
    begin
        exit(
            '
Dear ' + Invoice."Sell-to Customer Name" + ',
' +
            '
Please find attached invoice ' + Invoice."No." + ' ' +
            'dated ' + Format(Invoice."Posting Date", 0, '//') + '.
' +
            '
Amount due: ' + Format(Invoice.Amount, 0, '') + ' ' +
            Invoice."Currency Code" + '
' +
            '
Thank you for your business.
'
        );
    end;
}

Power Automate Triggers

For notifications to Teams, complex approval flows, or integrations with non-Microsoft systems, Power Automate is the right tool. BC fires Power Automate flows via two mechanisms: webhook subscriptions (real-time) and the Business Central connector (polling).

Triggering a Power Automate Flow from AL

codeunit 50505 "Power Automate Trigger"
{
    procedure TriggerFlow(FlowUrl: Text; Payload: JsonObject)
    var
        Client:   HttpClient;
        Request:  HttpRequestMessage;
        Response: HttpResponseMessage;
        Content:  HttpContent;
        Headers:  HttpHeaders;
        PayloadText: Text;
    begin
        Payload.WriteTo(PayloadText);
        Content.WriteFrom(PayloadText);
        Content.GetHeaders(Headers);
        Headers.Remove('Content-Type');
        Headers.Add('Content-Type', 'application/json');

        Request.Method := 'POST';
        Request.SetRequestUri(FlowUrl);
        Request.Content := Content;

        if not Client.Send(Request, Response) then
            Error('Power Automate trigger failed — network error.');

        if not Response.IsSuccessStatusCode() then
            Error('Flow returned HTTP %1. Check the flow run history.', Response.HttpStatusCode());
    end;

    procedure BuildInvoicePayload(Invoice: Record "Sales Invoice Header"): JsonObject
    var
        Payload: JsonObject;
    begin
        Payload.Add('invoiceNo',      Invoice."No.");
        Payload.Add('customerNo',     Invoice."Sell-to Customer No.");
        Payload.Add('customerName',   Invoice."Sell-to Customer Name");
        Payload.Add('amount',         Invoice.Amount);
        Payload.Add('postingDate',    Format(Invoice."Posting Date", 0, '--'));
        Payload.Add('postedBy',       UserId());
        exit(Payload);
    end;
}
💡
Store the Power Automate HTTP trigger URL in a setup table — never hardcode it in AL. Flows get recreated when moved between environments, and the URL changes. A setup table lets you update it without a code deployment.

Monitoring & Alerting

Automation that runs silently in the background must surface failures clearly. Build a simple monitoring layer from day one.

01

Job Queue Log Review

Check the Job Queue Log page daily for entries with status Error. Set up a Power Automate flow to send a Teams message whenever a Job Queue Entry fails — immediate visibility without manual checks.

02

Automation Log Table

Maintain a custom log table for every automated action: document number, action taken, timestamp, success/failure, error text. Give operations a page to search and filter without needing developer access.

03

Daily Summary Email

A Job Queue Entry running at 07:00 emails a summary of the previous day's automation: documents posted, notifications sent, failures. One email, full picture, zero manual checking.

04

Azure Application Insights

Enable Application Insights telemetry in your BC environment. Custom telemetry signals from AL (LogMessage) flow into your Azure workspace for dashboards, alerts, and trend analysis.

Automation Design Checklist

Event subscribers never perform slow operations synchronously — always enqueue
Every automated action is logged to a custom log table with timestamp and user
Job Queue failures surface to operations within minutes via Teams / email alert
Power Automate flow URLs and API keys stored in setup tables, not hardcoded
Approval workflow tested with delegation and substitution scenarios
Auto-posting codeunit is idempotent — re-running it after a crash doesn't double-post
Email notifications have a fallback for missing customer email addresses
All automation can be disabled per-company via a feature flag in setup
