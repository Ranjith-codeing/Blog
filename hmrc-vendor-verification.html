HMRC CIS Subcontractor Verification in Business Central

Before a UK construction contractor pays a subcontractor, HMRC decides how much tax to deduct. Here's how to ask them directly from AL — OAuth, the headers everyone forgets, and what to do with the answer.

R
Ranjith · May 2026

Under the UK Construction Industry Scheme (CIS), a contractor must verify each subcontractor with HMRC before the first payment. HMRC replies with a tax treatment that decides how much to withhold: nothing, 20%, or 30%. Get it wrong and the contractor is liable for the shortfall. Doing this by hand on the HMRC portal doesn't scale past a handful of subbies — so we wire it into Business Central. The integration is a textbook HMRC API job, which means OAuth 2.0 and the one thing that catches every first-timer: fraud-prevention headers.

CIS in one minute

CIS applies when a business pays subcontractors for construction work. The contractor deducts a percentage from the labour portion of each payment and pays it over to HMRC as an advance on the subcontractor's tax. The percentage depends entirely on the subcontractor's status with HMRC — which is exactly what verification establishes. Verification returns two things: a verification number and a tax treatment.

💡
You generally verify a subcontractor once. You only need to re-verify if you haven't included them in a CIS return in the current or previous two tax years. Cache the result on the vendor and you're not hammering the API.

The three deduction rates

Tax TreatmentDeductionWhen HMRC returns it
Gross0%Subcontractor has gross payment status — paid in full, settles tax later.
Net (Standard)20%Subcontractor is registered for CIS and matched.
Higher (Unmatched)30%Subcontractor could not be matched / is not registered.

The verification number is the same for every subcontractor verified in a single request — except for unmatched ones, where HMRC appends a one- or two-letter suffix. You store this number against the vendor and quote it on the CIS return.

OAuth 2.0 with HMRC

Every HMRC API uses OAuth 2.0. There are two access types, and CIS verification is user-restricted — it acts on behalf of a specific contractor, so a real person from that organisation authorises it once through Government Gateway. That consent yields a long-lived refresh token, which your integration then exchanges for short-lived access tokens going forward. 

codeunit 50700 "HMRC OAuth Manager"
{
    // Exchange the stored refresh token for a fresh access token.
    // The initial authorization-code consent is a one-time admin setup.
    procedure GetAccessToken(): Text
    var
        Setup: Record "HMRC CIS Setup";
        Client: HttpClient;
        Request: HttpRequestMessage;
        Response: HttpResponseMessage;
        Content: HttpContent;
        Headers: HttpHeaders;
        Json: JsonObject;
        Token: JsonToken;
        Body, ResponseText, RefreshToken: Text;
    begin
        Setup.Get();
        if not IsolatedStorage.Get('HMRC_RefreshToken', DataScope::Company, RefreshToken) then
            Error('HMRC has not been authorised yet. Run the consent wizard first.');

        Body :=
            'grant_type=refresh_token' +
            '&client_id=' + Setup."Client ID" +
            '&client_secret=' + GetClientSecret() +
            '&refresh_token=' + RefreshToken;

        Content.WriteFrom(Body);
        Content.GetHeaders(Headers);
        Headers.Remove('Content-Type');
        Headers.Add('Content-Type', 'application/x-www-form-urlencoded');

        Request.Method := 'POST';
        Request.SetRequestUri(Setup."Base URL" + '/oauth/token');
        Request.Content := Content;

        Client.Send(Request, Response);
        Response.Content.ReadAs(ResponseText);
        if not Response.IsSuccessStatusCode() then
            Error('HMRC token request failed (%1): %2', Response.HttpStatusCode(), ResponseText);

        Json.ReadFrom(ResponseText);
        // Persist the rotated refresh token, then return the access token
        if Json.Get('refresh_token', Token) then
            IsolatedStorage.Set('HMRC_RefreshToken', Token.AsValue().AsText(), DataScope::Company);
        Json.Get('access_token', Token);
        exit(Token.AsValue().AsText());
    end;
}
⚠️
HMRC rotates the refresh token on each use. If you don't persist the new one returned in the response, your next token request fails — a subtle bug that surfaces a few hours after go-live, not in testing.

Fraud-Prevention Headers

This is the part nobody expects. HMRC requires a set of Gov-Client-* and Gov-Vendor-* HTTP headers on every API call that describe the device and connection originating the request. They are mandatory and audited — calls without valid headers may be rejected or flagged, and HMRC publishes a "Test Fraud Prevention Headers" API specifically to validate them. 

local procedure AddFraudPreventionHeaders(var Headers: HttpHeaders)
var
    Setup: Record "HMRC CIS Setup";
begin
    Setup.Get();
    // Connection method for a server-side ERP integration
    Headers.Add('Gov-Client-Connection-Method', 'BATCH_PROCESS_DIRECT');
    Headers.Add('Gov-Client-Device-ID', Setup."Device ID");            // stable UUID per install
    Headers.Add('Gov-Client-User-IDs', 'bc=' + UserId());
    Headers.Add('Gov-Client-Timezone', 'UTC+00:00');
    Headers.Add('Gov-Client-Local-IPs', GetServerIp());
    Headers.Add('Gov-Vendor-Version', 'cis-connector=' + GetExtensionVersion());
    Headers.Add('Gov-Vendor-Product-Name', 'BC-CIS-Connector');
    Headers.Add('Gov-Vendor-License-IDs', '');
end;
💡
The exact header set depends on your connection method — a server batch process sends fewer headers than a browser-based app. Build them from one helper, version them with your extension, and run HMRC's validator API in your build pipeline so a header regression fails CI, not production.

The Verification Call

With a token and headers in hand, the verify request POSTs the contractor's references plus the subcontractor(s) to be checked. The body shape below is representative — always confirm the current contract on the HMRC Developer Hub, as field names and versions evolve. 

{
  "contractor": {
    "utr": "1234567890",
    "aoReference": "123PA12345678"
  },
  "subcontractors": [
    {
      "type": "soleTrader",
      "utr": "0987654321",
      "nino": "AA123456A",
      "firstName": "John",
      "lastName": "Doe"
    }
  ]
}
codeunit 50701 "CIS Verification"
{
    procedure VerifyVendor(var Vendor: Record Vendor)
    var
        OAuth: Codeunit "HMRC OAuth Manager";
        Setup: Record "HMRC CIS Setup";
        Client: HttpClient;
        Request: HttpRequestMessage;
        Response: HttpResponseMessage;
        Content: HttpContent;
        Headers: HttpHeaders;
        Payload: Text;
        ResponseText: Text;
    begin
        Vendor.TestField("CIS Subcontractor", true);
        Vendor.TestField("CIS UTR");
        Setup.Get();

        Payload := BuildVerifyPayload(Setup, Vendor);
        Content.WriteFrom(Payload);
        Content.GetHeaders(Headers);
        Headers.Remove('Content-Type');
        Headers.Add('Content-Type', 'application/json');

        Request.Content := Content;
        Request.GetHeaders(Headers);
        Headers.Add('Authorization', 'Bearer ' + OAuth.GetAccessToken());
        Headers.Add('Accept', 'application/vnd.hmrc.1.0+json');  // HMRC API versioning
        AddFraudPreventionHeaders(Headers);

        Request.Method := 'POST';
        Request.SetRequestUri(Setup."Base URL" +
            '/organisations/construction-industry-scheme/cis/verify');

        Client.Send(Request, Response);
        Response.Content.ReadAs(ResponseText);
        HandleResponse(Vendor, Response, ResponseText);
    end;
}

Handling the Response

A success returns the verification number and each subcontractor's treatment. Map it onto the vendor and stamp the date:

local procedure HandleResponse(var Vendor: Record Vendor;
                               var Response: HttpResponseMessage; ResponseText: Text)
var
    Json: JsonObject;
    Subs: JsonToken;
    Sub: JsonToken;
    Treatment, VerifyNo: JsonToken;
begin
    if Response.HttpStatusCode() = 401 then
        Error('HMRC rejected the token. Re-run authorisation.');
    if Response.HttpStatusCode() = 429 then
        Error('HMRC rate limit hit — back off and retry via the Job Queue.');
    if not Response.IsSuccessStatusCode() then
        Error('CIS verification failed (%1): %2', Response.HttpStatusCode(), ResponseText);

    Json.ReadFrom(ResponseText);
    Json.Get('subcontractors', Subs);
    foreach Sub in Subs.AsArray() do begin
        Sub.AsObject().Get('verificationNumber', VerifyNo);
        Sub.AsObject().Get('taxTreatment', Treatment);

        Vendor."CIS Verification No." := CopyStr(VerifyNo.AsValue().AsText(), 1, 20);
        Vendor."CIS Tax Treatment"   := MapTreatment(Treatment.AsValue().AsText());
        Vendor."CIS Verified On"     := Today();
        Vendor.Modify(true);
    end;
end;

local procedure MapTreatment(Code: Text): Enum "CIS Tax Treatment"
begin
    case Code of
        'gross':    exit("CIS Tax Treatment"::Gross);
        'net':      exit("CIS Tax Treatment"::Net);
        'unmatched','higher': exit("CIS Tax Treatment"::Higher);
    end;
end;
🚫
Never default an unknown response to 0%. If the treatment can't be mapped, leave the vendor unverified and surface an error. Under-deducting because of a parsing gap makes the contractor liable for the difference plus penalties.

Sandbox & Production

01

Test on the Sandbox

Develop against test-api.service.hmrc.gov.uk with HMRC-generated test users. Production is api.service.hmrc.gov.uk — store the base URL in setup so you flip environments without a code change.

02

Verify via Job Queue

Bulk-verify new subcontractors in the background with retry. Respect 429 with exponential back-off — HMRC throttles per application.

03

Protect the UTR

UTR and NINO are personal tax data. Classify the fields, restrict the page with a permission set, and keep the client secret in Isolated Storage.

04

Block Payment if Unverified

Subscribe to the payment-journal validation event and stop a CIS subcontractor payment going out at the wrong rate when no verification exists.

Readiness Checklist

Fraud-prevention headers validated against HMRC's test API in CI
Refresh token rotation persisted on every token exchange
Base URL configurable — sandbox vs production with no redeploy
Verification result cached on the vendor; re-verify only when required
Unknown tax treatment is an error, never a silent 0%
UTR / NINO classified and access-restricted
Payment blocked when a CIS subcontractor is unverified

Conclusion

HMRC verification is a compliance feature dressed as an integration. The REST mechanics are familiar — token, headers, POST, parse — but the consequences of cutting corners are financial, not cosmetic. Build the fraud-prevention headers properly, persist the rotating refresh token, fail loudly on anything you can't map, and cache the result so you're a good API citizen.

Key takeaways
  • CIS verification returns a verification number and a tax treatment of 0 / 20 / 30%.

  • Fraud-prevention headers are mandatory and audited — validate them with HMRC's test API.
  • Cache the result on the vendor and re-verify only when the rules require it.
  • An unmappable response is an error, never a silent 0% deduction.
