Performance Optimization Techniques in Business Central

From table key design and efficient SetFilter patterns to partial records, background processing, and profiling — a complete toolkit for writing AL that stays fast at scale.

Ranjith · March 2026

Performance issues in Business Central rarely appear in development. They show up three months after go-live, when the ledger table has two million rows and the month-end close report takes twelve minutes to open. This guide fixes those problems before they happen.

Table Key Design

Every filter you write in AL maps to a SQL WHERE clause. If that clause can't use an index, SQL Server scans every row in the table. At scale, unindexed filters on large tables are the single most common cause of BC performance problems.

Rules for Key Design

  • Keys should reflect your actual SetRange / SetFilter call patterns — look at the code first, then design the keys.
  • Put the most selective field first in a composite key — the field that narrows results the most.
  • Include SumIndexFields on keys used for CalcFields / CalcSums — this creates SQL computed columns instead of aggregating on the fly.
  • Avoid creating keys for every possible filter combination — each key has a write cost on insert/modify.
table 50400 "Invoice Upload Log"
{
    fields { ... }

    keys
    {
        // Clustered primary key — used for single-record Gets
        key(PK; "Entry No.") { Clustered = true; }

        // Used for: SetRange("Batch No."); SetRange(Status)
        // Most selective field first: Status has few values, so Batch No. leads
        key(BatchStatus; "Batch No.", Status) { }

        // Used for CalcSums on Amount — SumIndexFields avoids table scan
        key(CustomerDate; "Customer No.", "Posting Date")
        {
            SumIndexFields = Amount, "VAT Amount";
        }

        // Used for cleanup job: SetRange("Created At") for old record deletion
        key(CreatedAt; "Created At") { }
    }
}
⚠️
Avoid MaintainSiftIndex = false on keys with SumIndexFields unless you have a specific reason. It disables the SQL index and forces aggregation queries at read time.

Efficient Filter Patterns

The way you write filters in AL directly controls whether SQL uses an index or does a table scan. The difference can be orders of magnitude in query time.

SetRange vs SetFilter

// ✅ GOOD — SetRange maps directly to a SQL range scan (uses index)
Rec.SetRange("Customer No.", 'C001');
Rec.SetRange("Posting Date", DMY2Date(1,1,2026), DMY2Date(31,3,2026));

// ✅ GOOD — SetRange with a single value (equality, most efficient)
Rec.SetRange(Status, Status::Posted);

// ⚠️ CAREFUL — SetFilter with wildcards can prevent index use
Rec.SetFilter("Customer No.", 'C*');     // leading wildcard is OK
Rec.SetFilter("Customer No.", '*001');   // trailing wildcard → table scan

// ❌ BAD — Field expression in filter → always a table scan
Rec.SetFilter(Amount, '>%1', SomeCalculatedValue);
// Better: compute the threshold first, then use SetRange
MinAmount := ComputeMinAmount();
Rec.SetRange(Amount, MinAmount, MaxValue);

FindSet vs FindFirst vs Get

// ✅ Use Get() when you know the primary key — fastest possible lookup
if Rec.Get(PrimaryKeyValue) then ...

// ✅ Use FindFirst() when you only need the first match — adds TOP 1 to SQL
if Rec.FindFirst() then ...

// ✅ Use FindSet() for iterating multiple records
if Rec.FindSet() then
    repeat
        ...
    until Rec.Next() = 0;

// ❌ BAD — FindSet() when you only need to check existence
// This reads all matching rows into the cursor unnecessarily
if Rec.FindSet() then
    DoSomethingOnce();

// ✅ GOOD — Use IsEmpty() or FindFirst() for existence checks
if not Rec.IsEmpty() then ...

Partial Records

By default, FindSet() fetches every field in the table. If your table has BLOB fields, large text fields, or many columns you don't need, you're loading megabytes of data you'll never use. Partial records let you specify exactly which fields to load. 

// ❌ Without partial records — loads ALL fields including large BLOBs
Rec.SetRange(Status, Status::Pending);
if Rec.FindSet() then
    repeat
        TotalAmount += Rec.Amount;
    until Rec.Next() = 0;

// ✅ With partial records — only loads the fields you specify
Rec.SetLoadFields(Amount, Status, "Customer No.");
Rec.SetRange(Status, Status::Pending);
if Rec.FindSet() then
    repeat
        TotalAmount += Rec.Amount;
    until Rec.Next() = 0;
💡
SetLoadFields is particularly powerful when processing large tables in background jobs. A loop over 100,000 rows loading 5 fields instead of 40 can be 4–8x faster depending on field sizes.

CalcFields Discipline

// ❌ BAD — CalcFields inside a loop issues N SQL queries
if Rec.FindSet() then
    repeat
        Rec.CalcFields("Balance (LCY)"); // 1 extra SQL query per row
        if Rec."Balance (LCY)" > 0 then ...
    until Rec.Next() = 0;

// ✅ GOOD — Use a query object or SumIndexFields key instead
// Or filter before the loop to reduce the working set
Rec.SetRange("Balance (LCY)", 0.01, 999999999);

Loop & Commit Patterns

Long-running loops that process thousands of records have two enemies: lock escalation and transaction log bloat. Strategic use of Commit() and row locking controls both. 

// Pattern: process in chunks with periodic Commit()
codeunit 50401 "Bulk Processor"
{
    trigger OnRun()
    var
        Rec:       Record "Large Table";
        Counter:   Integer;
        ChunkSize: Integer;
    begin
        ChunkSize := 500; // commit every 500 rows
        Counter := 0;

        Rec.SetRange(Status, Status::Pending);
        Rec.SetLoadFields("Entry No.", Status, Amount);

        if not Rec.FindSet(true) then exit; // true = lock rows for update

        repeat
            ProcessRecord(Rec);
            Counter += 1;

            if Counter mod ChunkSize = 0 then
                Commit(); // release locks, flush transaction log

        until Rec.Next() = 0;

        Commit(); // final commit for remainder
    end;
}
⚠️
Commit() inside a loop means failures after a commit cannot be rolled back. Design your processing to be re-entrant: mark records as processed before committing so a crash doesn't reprocess them on restart.

Background Processing

Any operation taking more than 2–3 seconds should run in the background. Business Central's Job Queue is the primary tool, but Background Sessions offer a lighter option for in-page async work.

Background Sessions

// Spawn a background session from a page action
// Runs in a separate server session — doesn't block the UI
var
    SessionId: Integer;
begin
    if not StartSession(SessionId, Codeunit::"Heavy Calculation Job", CompanyName()) then
        Error('Could not start background session.');

    Message('Processing started in background. Session: %1', SessionId);
end;

Profiling with AL Profiler

Guessing where the bottleneck is wastes time. The AL Profiler in VS Code shows exactly how many milliseconds each AL method and SQL statement takes — use it before every optimisation effort.

How to Profile

  1. Open your BC extension in VS Code
  2. Open the Command Palette: Ctrl+Shift+P
  3. Run AL: Start Profiling — choose your environment
  4. Perform the slow action in BC
  5. Run AL: Stop Profiling — the flame graph opens automatically

Reading the Flame Graph

  • Wide bars = slow code — the wider, the more wall-clock time spent there
  • Look for NavCommon.dll / SQL frames — these reveal database round-trips
  • Repeated narrow stacks appearing hundreds of times = a loop with per-iteration DB calls
  • A single very wide bar is usually a single slow query — add a key to fix it
💡
Also check Database Calls in the BC Performance Profiler (available under Tools > Performance Profiler in the web client). It shows the exact SQL statements issued, row counts, and durations — invaluable for diagnosing missing indexes.

Page & Query Performance

Use Query Objects for Aggregations

When a page or report needs aggregated data (sums, counts, averages across related tables), a Query object is significantly faster than iterating records in AL:

query 50400 "Customer Invoice Summary"
{
    QueryType = Normal;

    elements
    {
        dataitem(SalesInvoiceHeader; "Sales Invoice Header")
        {
            column(CustomerNo;   "Sell-to Customer No.") { }
            column(PostingDate;  "Posting Date")         { }
            column(InvoiceCount; "No.")
            {
                Method = Count;
            }
            column(TotalAmount;  Amount)
            {
                Method = Sum;
            }
        }
    }
}

FlowField Discipline on List Pages

FlowFields on list pages calculate for every visible row simultaneously. A list with 20 rows and 3 FlowFields issues 60 SQL queries on open. Mitigations:

  • Set Enabled = false on FlowField columns that aren't shown by default
  • Use AutoCalcField = false and calculate on demand (drill-down action)
  • Replace FlowFields with Query-backed FactBoxes where possible

Common Anti-Patterns

01

N+1 Query Problem

Calling Get() or CalcFields() inside a FindSet loop. Each iteration hits the database. Fix: pre-fetch with a Query or join, or refactor to a single-pass approach.

02

Unfiltered FindSet on Large Tables

Calling FindSet() with no filters on Item Ledger Entries or Value Entries. Always apply at least one indexed filter before iterating. Never assume a table is small.

03

Missing Commit in Bulk Jobs

Processing 50,000 records in a single transaction holds locks on all modified rows for the entire duration. Other sessions block. Add periodic Commit() calls every 500–1000 rows.

04

Modifying Records Unnecessarily

Calling Modify() even when nothing changed. Always check if a field actually needs updating before writing. Unnecessary writes bloat the transaction log and trigger triggers.
