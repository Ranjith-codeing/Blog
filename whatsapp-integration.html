WhatsApp Notifications from Business Central

Customers don't check email inboxes the way they check WhatsApp. Here's how to push posted-invoice alerts — and any other notification — straight from AL using Meta's WhatsApp Business Cloud API.

R
Ranjith · May 2026

Email notifications get buried; WhatsApp messages get read in minutes. For a lot of Business Central customers — especially in SMB and retail — a WhatsApp invoice alert is the difference between "paid this week" and "chasing next month". The good news: Meta's WhatsApp Business Cloud API is a plain REST endpoint, so everything you need lives inside an AL HttpClient call. This walkthrough builds it from a setup table up to an automatic send when a sales invoice is posted.

Why WhatsApp, and what the Cloud API gives you

The WhatsApp Business Cloud API is hosted by Meta — there's no on-prem WhatsApp server to run (that was the old "On-Premises API", now deprecated). You register an app, get a Phone Number ID and an access token, and POST JSON to graph.facebook.com. Two message types matter for ERP notifications:

  • Text messages — free-form, but only allowed inside a 24-hour customer service window that opens when the customer last messaged you.
  • Template messages — pre-approved, parameterised messages you can send any time. This is what you want for unsolicited invoice alerts.
⚠️
The 24-hour rule trips everyone up. You cannot send a free-form text message to a customer who hasn't messaged you in the last 24 hours — Meta will reject it. Business-initiated notifications like "your invoice is ready" must use an approved template.

Solution Architecture

Business Central never talks to WhatsApp directly — it talks to the Meta Graph API, which delivers the message. Three moving parts:

🏢
Business Central
Posting event, message body, recipient lookup
→
🌐
Meta Graph API
graph.facebook.com · Bearer token · JSON
→
💬
WhatsApp
Message delivered to the customer's phone

Meta Prerequisites

Set these up once on the Meta side before writing AL:

  1. Create a Meta for Developers app and add the WhatsApp product.
  2. Connect a WhatsApp Business Account (WABA) and register a phone number — you'll get a Phone Number ID.
  3. Generate a permanent access token via a System User (the temporary 24-hour token is fine for testing only).
  4. Create and submit a message template for approval — for this article, one named sales_invoice_cvt with five body parameters.
💡
The messages endpoint is keyed by the Phone Number ID, not the WABA ID. It's an easy mix-up — if you get a (#100) Param ... is not valid error, this is the first thing to check.

Setup Table & Page

A single-record setup table holds the connection configuration. The token is sensitive, so flag it accordingly and mask it on the page.

table 57100 "WA Setup"
{
    Caption = 'WhatsApp Setup';
    DataClassification = CustomerContent;

    fields
    {
        field(1; "Primary Key"; Code[10]) { Caption = 'Primary Key'; }
        field(2; "API Version"; Text[10])
        {
            Caption = 'API Version';   // e.g. v20.0
            InitValue = 'v20.0';
        }
        field(3; "Phone Number ID"; Text[50]) { Caption = 'Phone Number ID'; }
        field(4; "Access Token"; Text[250])
        {
            Caption = 'Access Token';
            ExtendedDatatype = Masked;   // hide on the card
        }
        field(5; "Country Code"; Text[5])
        {
            Caption = 'Default Country Code';
            InitValue = '91';
        }
        field(6; Enabled; Boolean) { Caption = 'Enabled'; }
    }

    keys { key(PK; "Primary Key") { Clustered = true; } }
}

The setup card adds a Test Message action so an administrator can confirm the connection without posting anything:

page 57100 "WA Setup"
{
    Caption = 'WhatsApp Setup';
    PageType = Card;
    SourceTable = "WA Setup";
    ApplicationArea = All;
    UsageCategory = Administration;
    InsertAllowed = false;
    DeleteAllowed = false;

    layout
    {
        area(Content)
        {
            group(General)
            {
                field("Phone Number ID"; Rec."Phone Number ID") { ApplicationArea = All; }
                field("API Version"; Rec."API Version") { ApplicationArea = All; }
                field("Access Token"; Rec."Access Token")
                {
                    ApplicationArea = All;
                    ExtendedDatatype = Masked;
                }
                field("Country Code"; Rec."Country Code") { ApplicationArea = All; }
                field(Enabled; Rec.Enabled) { ApplicationArea = All; }
            }
        }
    }

    actions
    {
        area(Processing)
        {
            action(TestMessage)
            {
                ApplicationArea = All;
                Caption = 'Send Test Message';
                Image = SendTo;
                trigger OnAction()
                var
                    WA: Codeunit "WhatsApp Integration";
                begin
                    WA.SendTextMessage('This is a test message from Business Central.',
                                       Rec."Country Code" + '9943076061');
                end;
            }
        }
    }

    trigger OnOpenPage()
    begin
        if not Rec.Get() then begin
            Rec.Init();
            Rec.Insert();
        end;
    end;
}
🔒
Hardening note: a masked text field still stores the token in the database. For production, keep the token in Isolated Storage and expose only a write-only "set token" action — see Production Hardening below.

Sending a Text Message

The core of the integration is one procedure: build the JSON body, attach the Bearer token, POST it. The WhatsApp payload for a text message nests the message under a text object: 

codeunit 57101 "WhatsApp Integration"
{
    procedure SendTextMessage(Body: Text; ToPhone: Text): Boolean
    var
        Setup: Record "WA Setup";
        Client: HttpClient;
        Request: HttpRequestMessage;
        Response: HttpResponseMessage;
        Headers: HttpHeaders;
        Content: HttpContent;
        Payload: JsonObject;
        TextObj: JsonObject;
        ResponseText: Text;
    begin
        Setup.Get();
        Setup.TestField(Enabled);
        Setup.TestField("Phone Number ID");

        // { "messaging_product": "whatsapp", "to": "...",
        //   "type": "text", "text": { "preview_url": true, "body": "..." } }
        Payload.Add('messaging_product', 'whatsapp');
        Payload.Add('to', ToPhone);
        Payload.Add('type', 'text');
        TextObj.Add('preview_url', true);
        TextObj.Add('body', Body);
        Payload.Add('text', TextObj);

        SendToGraph(Setup, Payload, Response);

        Response.Content.ReadAs(ResponseText);
        if not Response.IsSuccessStatusCode() then
            Error('WhatsApp send failed (%1): %2', Response.HttpStatusCode(), ResponseText);
        exit(true);
    end;

    local procedure SendToGraph(Setup: Record "WA Setup"; Payload: JsonObject;
                                var Response: HttpResponseMessage)
    var
        Client: HttpClient;
        Request: HttpRequestMessage;
        Content: HttpContent;
        Headers: HttpHeaders;
        Body: Text;
    begin
        Payload.WriteTo(Body);
        Content.WriteFrom(Body);

        Content.GetHeaders(Headers);
        Headers.Remove('Content-Type');
        Headers.Add('Content-Type', 'application/json');

        Request.Content := Content;
        Request.GetHeaders(Headers);
        Headers.Add('Authorization', 'Bearer ' + GetToken(Setup));

        Request.Method := 'POST';
        Request.SetRequestUri(StrSubstNo('https://graph.facebook.com/%1/%2/messages',
                                         Setup."API Version", Setup."Phone Number ID"));
        Client.Send(Request, Response);
    end;
}
💡
Notice the Content-Type header is set on the content, while Authorization is set on the request. AL splits headers between the two — a classic source of "why is my header missing" bugs.

Template Messages

For invoice alerts you need a template, because the customer hasn't opened a service window. A template message references the approved template by name and language, then supplies an ordered array of parameters that fill the {{1}}, {{2}}… placeholders in the template body. 

procedure SendInvoiceTemplate(SalesInvHeader: Record "Sales Invoice Header"; ToPhone: Text)
var
    Setup: Record "WA Setup";
    Response: HttpResponseMessage;
    Payload: JsonObject;
    Template: JsonObject;
begin
    Setup.Get();
    Setup.TestField(Enabled);

    Payload.Add('messaging_product', 'whatsapp');
    Payload.Add('recipient_type', 'individual');
    Payload.Add('to', ToPhone);
    Payload.Add('type', 'template');

    BuildInvoiceTemplate(Template, SalesInvHeader);
    Payload.Add('template', Template);

    SendToGraph(Setup, Payload, Response);
    if not Response.IsSuccessStatusCode() then
        LogFailure(SalesInvHeader."No.", ToPhone, Response);
end;

local procedure BuildInvoiceTemplate(var Template: JsonObject;
                                     SalesInv: Record "Sales Invoice Header")
var
    Language: JsonObject;
    Components: JsonArray;
    BodyComponent: JsonObject;
    Parameters: JsonArray;
begin
    Template.Add('name', 'sales_invoice_cvt');
    Language.Add('code', 'en');
    Template.Add('language', Language);

    // Body parameters, in the exact order they appear in the approved template
    AddTextParam(Parameters, SalesInv."Sell-to Customer Name");
    AddTextParam(Parameters, SalesInv."External Document No.");
    AddTextParam(Parameters, SalesInv."No.");
    AddTextParam(Parameters, Format(SalesInv.Amount, 0, ''));
    AddTextParam(Parameters, Format(SalesInv."Due Date", 0, '//'));

    BodyComponent.Add('type', 'body');
    BodyComponent.Add('parameters', Parameters);
    Components.Add(BodyComponent);
    Template.Add('components', Components);
end;

local procedure AddTextParam(var Parameters: JsonArray; Value: Text)
var
    Param: JsonObject;
begin
    Param.Add('type', 'text');
    Param.Add('text', Value);
    Parameters.Add(Param);
end;
⚠️
Parameter order is everything. WhatsApp matches parameters positionally, not by name. If you reorder the body parameters here without changing the approved template, customers get "Invoice 4,500.00 due to John Smith" — change the template and the code together.

Auto-Send on Posting

The whole point is zero manual effort. Subscribe to the sales-post event, look up the posted invoice and the customer's phone, and fire the template. Keep the subscriber thin and never let a messaging failure roll back the posting. 

codeunit 57102 "WA Posting Subscriber"
{
    [EventSubscriber(ObjectType::Codeunit, Codeunit::"Sales-Post",
                     'OnAfterFinalizePosting', '', false, false)]
    local procedure OnAfterFinalizePosting(
        var SalesHeader: Record "Sales Header";
        var SalesInvoiceHeader: Record "Sales Invoice Header")
    var
        WA: Codeunit "WhatsApp Integration";
        Customer: Record Customer;
        Setup: Record "WA Setup";
        Phone: Text;
    begin
        if SalesInvoiceHeader."No." = '' then exit;
        if not Setup.Get() or not Setup.Enabled then exit;
        if not Customer.Get(SalesInvoiceHeader."Sell-to Customer No.") then exit;
        if Customer."Phone No." = '' then exit;

        SalesInvoiceHeader.CalcFields(Amount);
        Phone := BuildPhone(Setup."Country Code", Customer."Phone No.");

        // Never block or roll back posting because a notification failed
        if not TrySend(WA, SalesInvoiceHeader, Phone) then
            LogLastError(SalesInvoiceHeader."No.");
    end;

    [TryFunction]
    local procedure TrySend(var WA: Codeunit "WhatsApp Integration";
                            SalesInv: Record "Sales Invoice Header"; Phone: Text)
    begin
        WA.SendInvoiceTemplate(SalesInv, Phone);
    end;
}
💡
Wrapping the send in a [TryFunction] is the key safety move: if Meta is down or the number is invalid, the invoice still posts cleanly and the failure is logged for retry — exactly the pattern from Automating Document Processing.

Production Hardening

01

Token in Isolated Storage

Move the access token out of the table into IsolatedStorage (DataScope::Company). Expose only a write-only "Set Token" action — admins can update it, nobody can read it back.

02

Send via Job Queue

For volume, enqueue each send instead of calling Graph synchronously inside the posting transaction. Faster posting, natural retry, no user-visible latency.

03

Phone Normalisation

WhatsApp wants E.164 digits with no +, spaces, or leading zeros. Strip and prefix the country code centrally so one bad contact record can't break a send.

04

Delivery Log & Webhook

Persist every attempt (message ID, status, error). Subscribe to Meta's status webhook to capture delivered / read / failed for a full audit trail.

Readiness Checklist

Permanent System User token — not the temporary 24-hour token
Token stored in Isolated Storage, never returned to the UI
Business-initiated alerts use an approved template, not free-form text
Phone numbers normalised to E.164 before sending
Sends wrapped in a TryFunction so posting never rolls back
Every attempt logged; status webhook captures delivery state
Customer opt-in recorded — required by WhatsApp Business policy

Conclusion

A WhatsApp notification layer is one setup table, one codeunit, and one event subscriber — but the details (the 24-hour window, template parameter order, token security) are what separate a demo from something you'd run for a paying customer. Start with the test action, get one template approved, then wire it into posting.

✅
What's next? The same Graph-API-from-AL pattern powers the HMRC CIS Verification integration — different endpoint, same HttpClient discipline.
Key takeaways
  • Use the hosted Cloud API (graph.facebook.com) — no server to run.
  • Business-initiated alerts must use an approved template; free-form text needs an open 24-hour window.
  • Template parameters are positional — keep code and approved template in lockstep.
  • Auto-send from a posting subscriber, wrapped in a TryFunction so messaging never blocks posting.
  • Keep the token in Isolated Storage and log every delivery.
